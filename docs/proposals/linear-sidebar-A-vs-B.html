Linear tickets in the Orchestra sidebar — two options, side by side

Both mockups use Orchestra's real sidebar palette, spacing and row geometry (lifted from src/renderer/styles.css), with realistic data: an orchestrator coordinating spawned agents, worktrees carrying NMC-* branch badges, and NMC tickets that have no worktree yet.

◆ The question: should a Linear ticket be able to appear in the sidebar before anyone starts work on it?
Option A · ticket-as-badge

Tickets attach to work that already exists

The sidebar keeps showing only workspaces. The CLI lets you pin a ticket to a workspace whose branch name doesn't spell out the issue key — it then shows the same badge as today.

These never appear anywhere in Option A. A ticket only becomes visible once somebody has already created a workspace for it. Your backlog stays in Linear; Orchestra keeps showing only work in flight.
▲ No standalone ticket entry. The sidebar looks essentially unchanged. The only visible difference from today is that a workspace whose branch doesn't mention the issue can now carry a badge anyway — like refactor-wp-payload-builder above.

What you gain / lose

  • +Ships in well under an hour; zero risk to the sidebar you use all day.
  • +Fixes the real gap that a branch name is the only way to link a ticket today.
  • +Nothing new to learn — it's the badge you already recognise.
  • −Your backlog stays invisible. You still switch to Linear to see what's next.
  • −No way to go from "here's a ticket" to "spawn an agent on it" inside Orchestra.
The one-line tradeoff Cheap and safe, but it only labels work you already started — it does not put your backlog in front of you.
Option B · ticket-as-row

Tickets become a queue you can start work from

A new Tickets section sits alongside Orchestrators and Scratch, listing Linear issues as rows of their own — before any worktree exists. Each row can spawn an agent on the spot.

The ticket leaves the Tickets section and becomes an ordinary workspace row, with today's branch-derived badge taking over. A ticket is never shown twice — the queue only ever holds work that hasn't started.
◆ The diamond is deliberate. Ticket rows use a hollow diamond, not the round status dot — so a ticket never reads as an agent that is running or waiting. The section disappears entirely when you have no tickets pinned, so nothing changes for anyone who doesn't use it.

What you gain / lose

  • +Your next work is visible next to your running agents — no tab switch.
  • +One click from "a ticket exists" to "an agent is working on it".
  • +Includes everything Option A does — pinning still works, via graduation.
  • −Roughly 8× the code; touches the saved data file and the sidebar you use daily.
  • −More to keep correct: ticket state must refresh from Linear, and can go stale.
The one-line tradeoff Costs meaningfully more and touches the sidebar, but it's the only option that puts work you haven't started yet in front of you.
How to read this: the two sidebars are drawn with Orchestra's actual colours, row heights and badge shapes, so the visual difference here is the difference you'd see in the app. Everything above the Tickets section is byte-identical between the two — Option B adds a section, it does not rearrange what's already there.
